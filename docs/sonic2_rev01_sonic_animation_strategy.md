# Sonic 2 (Genesis) REV01 (Sonic) animation scripts (frame indices) and $FF/$FE/$FD delay semantics

This is a ROM first implementation strategy for a Sonic 2 REV01 player animation interpreter (Sonic), with a concrete mapping from animation IDs (states) to the frame indices (mapping_frame values) used by each script.

The intent is that your engine loads and runs the same animation scripts as the original game (byte accurate behavior, including $FF/$FE/$FD start flags and $FF/$FE/$FD end flags).

## What “frame index” means in Sonic 2

Sonic 2’s player animation scripts output a single byte called mapping_frame (the “sprite number” in the disassembly). That byte selects a sprite mapping frame for Sonic’s current pose (and, via the DPLC, the correct art tiles). It is not a tile index, and it is not a pattern ID.

## ROM anchors (REV01)

These are disassembly label comments that map to ROM addresses:

Sonic animation script pointer table: ; off_1B618:.
Interpreter entry for fixed delay scripts: ; loc_1B384:.
End flag handling ($FF/$FE/$FD and $F0 to $FC): ; loc_1B3D4:.
Walk and run special timing (start flag $FF): ; loc_1B3F4:.
Dynamic pattern loader entry (mapping_frame to DPLC): ; loc_1B848:.

For REV01, you want the gameRevision<>3 content of SonicAniData (not the KiS2 conditional block).

## Script format (as stored in ROM)

Each animation script is a byte stream:

The first byte is either a fixed delay (00 to 7F) or a start flag (FF, FE, FD).
The following bytes are mapping_frame values.
The first byte in the frame stream that is >= $F0 is treated as an end flag.

End flags and parameters:
$FF ends the script and restarts it from anim_frame 0.
$FE ends the script and performs a loopback (one extra parameter byte follows, which is a byte distance).
$FD ends the script and switches to another animation ID (one extra parameter byte follows, which is the new anim value).
Any other end byte in the range $F0 to $FC returns immediately (the current mapping_frame remains whatever was last set).

## Correct handling of $FF, $FE, $FD delay semantics (REV01)

There are two different “$FF/$FE/$FD semantics” you must implement correctly (start flags at the beginning of the script, and end flags inside the script). They are distinct.

### Start flag semantics (first byte of the script)

If the first byte is 00 to 7F, it is a fixed delay. The interpreter decrements anim_frame_duration each tick. When it becomes negative, it reloads anim_frame_duration from the delay byte and advances anim_frame by one.

If the first byte is 80 to FF, the engine uses special timing logic. In REV01’s Sonic scripts, this is used as three explicit start flags:

$FF means WalkRun timing (variable speed based on inertia, plus angle related offsets for slopes and similar cases).
$FE means Roll timing (variable speed based on inertia, with a choice between Roll and Roll2 scripts based on speed).
$FD means Push timing (variable speed based on inertia).

The original 68000 control flow is:

SAnim_Do reads the first byte and branches to SAnim_WalkRun if it is negative.
SAnim_WalkRun checks for $FF by adding 1 to the byte and comparing to zero.
If not $FF, it runs the Roll handler.
The Roll handler checks for $FE by adding 1 and comparing to zero.
If not $FE, it runs the Push handler (which corresponds to $FD).

If you want maximum fidelity, mirror this structure exactly (meaning an unexpected start flag value should follow the same branching behavior, even if your scripts never use it).

### End flag semantics (first >= $F0 byte in the frame stream)

When the current sprite byte read from the script is >= $F0, REV01 treats it as an end flag.

$FF restarts the animation by setting anim_frame to 0, then immediately reading the first sprite byte again.
$FE loops back by a byte distance given by the next parameter byte (the interpreter subtracts that value from anim_frame and from its internal offset, then reads the sprite at the new position).
$FD sets anim to the parameter byte.
Other values in $F0 to $FC return (no restart, no loop, no switch).

This is why you must not treat $FE or $FD as simple one byte terminators (both consume an extra parameter byte).

## Implementation strategy (engine reimplementation)

This assumes you are reading REV01 ROM data (raw) and are re running the original script logic rather than hardcoding poses.

### State variables (match original intent)

Sonic uses these per object fields (names match disassembly conventions):

anim (current animation ID).
prev_anim (previous animation ID).
anim_frame (byte index into the script’s frame stream, starting at 0).
anim_frame_duration (countdown timer for fixed delay scripts, also reused as a variable duration accumulator in the start flag handlers).
mapping_frame (the “frame index” you ultimately use to select mappings and DPLC).

Also relevant inputs for the special start flag handlers include inertia, angle, status bits, and Super_Sonic_flag (plus environment dependent checks).

### Update pseudocode (faithful shape)

This is intentionally close to the 68000 control flow, while still being readable.

```text
a1 = (Super_Sonic_flag ? SuperSonicAniData : SonicAniData)   (REV01)

if anim != prev_anim:
  prev_anim = anim
  anim_frame = 0
  anim_frame_duration = 0
  clear status bit 5

script_ptr = a1 + word(a1 + anim*2)  (offsetTableEntry.w semantics)
delay_or_flag = byte(script_ptr + 0)

if delay_or_flag is negative:
  run the special handler for $FF/$FE/$FD start flags
  (this sets mapping_frame directly and updates anim_frame and anim_frame_duration)
  return

anim_frame_duration -= 1
if anim_frame_duration >= 0:
  return

anim_frame_duration = delay_or_flag
sprite = byte(script_ptr + 1 + anim_frame)

if sprite >= $F0:
  handle end flag ($FF restart, $FE loopback param, $FD set anim param, otherwise stop)
  return

mapping_frame = sprite
anim_frame += 1
```
